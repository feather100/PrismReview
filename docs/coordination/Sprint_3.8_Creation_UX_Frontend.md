# Sprint 3.8 — Review Creation UX Polish (Frontend)

## 1. 概览

本次 Sprint 在未进行后端改动与未接写真实智能体的前提下，针对“创建真实评审”这一核心业务链条，完成了前端向的引导式体验（UX Polish）升级，消除了用户在创建过程中的困惑感。

## 2. 交互变化明细

### 2.1 `/reviews/new` 表单引导增强
- **视觉层**：在“评审材料”输入框下方的帮助信息中，醒目地补充了橙黄色的警告：`当前支持文本/Markdown 粘贴，附件上传暂未接入。`。
- **反馈层**：提交成功后的 Toast 提示由机械的 `评审创建成功！` 改为了带有行进感的 `评审已创建，正在进入诊断页。`，与路由跳转的连贯性更强。

### 2.2 `DiagnosisPage` 初始向导 (Draft 态体验翻新)
- **视觉层重构**：彻底废弃了原先易被误认为“数据没查出来”的 `Empty` 占位组件。
- **引导式 UI**：引入了 `antd` 的 `Steps` 组件。现在当用户带着刚刚创建好的材料首次进入诊断页时，会看到一个宽阔大气的启动卡片：
  - **大标题**：`评审材料已提交，开始您的 AI 架构评审之旅`
  - **路线图**：清晰展示四大里程碑 (`开始诊断` → `确认评审团` → `进入会议` → `查看报告`)，且当前高亮聚焦于第一步，有效降低了业务黑盒带来的门槛。

### 2.3 “我的评审”列表项主操作按状态推演
为了让用户的注意力集中在“当前我能做什么”上，我动态化了列表内第一个 `Button` 的文本，而不是机械地保持“查看诊断”：
- **`draft`（草稿）**：按钮变为高亮的 **`开始诊断`**。
- **`ready`（待评审）**：按钮变为高亮的 **`确认评审团`**。
- 只有在其后状态才退化为基础的 `查看诊断`。原有的 `进入会议室` / `查看报告` 按钮激活规则与禁用提示仍保持严格的业务语义。
