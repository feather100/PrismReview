# Sprint 3.10 — Review Detail Timeline (Frontend)

## 1. 概览

根据 `Sprint 3.9` 后端关于时间线契约的结论分析，本次 Sprint 在无需后端新增 API 且不改动任何数据结构的前提下，通过前端组合现有状态字段，在详情诊断页成功实现了「全周期透明可视」的时间线模块（Timeline UI）。

## 2. 推导规则与实现细节

时间线数据严格基于 `GET /api/reviews/{id}` 返回的 `status` 以及 `GET /api/reviews/{id}/diagnosis` 返回的诊断数据存在与否进行计算推断。

**五大时间线锚点**：
1. **创建评审** (固化为已完成)
2. **开始诊断**
3. **确认评审团**
4. **进入评审会议**
5. **生成评审报告**

**推断矩阵表**：

| 当前 `status` | 是否有诊断结果 | 映射的当前执行步骤 (0-indexed) | 备注 (步骤状态 `StepsStatus`) |
| :--- | :--- | :--- | :--- |
| `draft` | 无 (或正在请求中) | `1` (开始诊断) | `process`，提示用户点击开始 |
| `ready` | 有 | `2` (确认评审团) | `process`，提示用户确认并进入会议 |
| `running` (含中断、总结) | 有 | `3` (进入评审会议) | `process`，正在流式打字中 |
| `completed` | 有 | `4` (生成评审报告) | `finish`，全链路点亮 |
| `failed` | - | 取决于是否有诊断数据 | `error`，时间线中止并高亮错误态 |

**下一步动作提示 (`nextActionText`)** 逻辑：
- `draft` → “下一步：开始诊断”
- `ready` → “下一步：确认评审团”
- `running` 等 → “下一步：等待会议完成”
- `completed` → “下一步：查看评审报告”
- `failed` → “评审失败，请查看错误信息”

## 3. 视觉与架构重构

### `apps/web/src/features/diagnosis/DiagnosisPage.tsx`
- **提权与统管**：将原本只挂在 `draft` 空态下孤立的 `Steps` 组件，重构提升为独立渲染模块 `timelineModule`。
- **融合布局**：新版 Timeline 以 `Card` 的形式稳定地驻留在 DiagnosisPage 的页面标题下方、实际内容之上（即页面次顶层）。
- **兼容空态**：即使 `data` 不存在（即初次草稿态），这套统一的模块依然无缝衔接。并且原有专门用于展示 `draft` 的迎宾区域代码也依据新布局做出了适当精简（移除了硬编码的4步组件）。

## 4. 验证情况

- `apps/web tsc` 检查：**0 errors** ✅
- **流转验证**：
  - 新建 review 落在 draft，顶部渲染出步骤，当前指向第二步 `开始诊断`，符合预期。
  - 成功请求诊断后状态变更为 `ready`，时间线步进至第三步 `确认评审团`，符合预期。
  - 完成开会至 `completed` 后，时间线五步全部点亮为绿色/蓝色完成态，符合预期。
